Infrastructure Decisions
1 Configurable Ports via Environment Variables Phase 1 ✓ WORKING
⚠ Before — Problem
Ports hardcoded in application.yml (e.g. server.port: 8081). Changing any port requires a source code edit and rebuild. Running alongside other servers causes Address already in use errors with no quick fix.
✓ After — Solution
Every port uses ${ENV_VAR:default} Spring Boot placeholder. Override at runtime via environment variable, .env file, or -Dserver.port=XXXX JVM arg. Zero code changes needed. Example: set ACCOUNT_SERVICE_PORT=9081 then start.
Services:All 6
Service env vars:CONFIG_SERVER_PORT (8888), REGISTRY_PORT (8761), GATEWAY_PORT (8090), ACCOUNT_SERVICE_PORT (8081), TRANSACTION_SERVICE_PORT (8082), NOTIFICATION_SERVICE_PORT (8083)
Infra env vars:KAFKA_PORT (9092), ZOOKEEPER_PORT (2181), KAFDROP_PORT (9000)
2 bootstrap.yml for Config Clients (Spring Cloud 2020+ Requirement) Phase 1 ✓ WORKING
⚠ Before — Problem
In Spring Cloud 2020+, bootstrap.yml is not loaded by default. Services start with wrong config values — silently use application.yml defaults instead of fetching real values from config-server. Config properties load too late in the application context lifecycle.
✓ After — Solution
spring-cloud-starter-bootstrap dependency added explicitly to each client service POM. Each client has a bootstrap.yml with spring.cloud.config.uri. Config is fetched before the main application context loads — correct values always used.
Services:api-gateway, account-service, transaction-service, notification-service
Key dependency:spring-cloud-starter-bootstrap (must be declared explicitly in Spring Cloud 2021.x)
Key config:spring.cloud.config.uri: ${CONFIG_SERVER_URI:http://localhost:${CONFIG_SERVER_PORT:8888}}
3 Config Server Native Profile (Classpath-Based Config, No Git) Phase 1 ✓ WORKING
⚠ Before — Problem
Spring Cloud Config Server defaults to a Git backend. Without an explicit profile, it tries to clone a remote Git repo on startup. Fails with CannotCreateTransactionException or JGitInternalException if no Git remote is configured. Requires internet access just to run locally.
✓ After — Solution
spring.profiles.active=native set in config-server application.yml. Config files served from classpath:/configurations/ — no Git, no network. Per-service files: account-service.yml, transaction-service.yml, etc. Works fully offline. Config file name = spring.application.name of the client.
Services:config-server only
Config files served:config-server/src/main/resources/configurations/{service-name}.yml
Key property:spring.cloud.config.server.native.search-locations: classpath:/configurations/
4 Config Client Retry on Slow Config-Server Startup Phase 1 ✓ WORKING
⚠ Before — Problem
Config-server takes several seconds to start (JVM + Spring context init). If account-service or api-gateway start simultaneously, they connect once, fail immediately with Connection refused, and crash. Requires manual startup delays or ordering in scripts — fragile.
✓ After — Solution
spring.cloud.config.fail-fast=true + spring-retry + spring-boot-starter-aop added to each client service. Services retry up to 6 times with exponential backoff (1s initial, ×1.1 multiplier, max 2s). Config-server has ~10s window to become ready — sufficient for local and Docker startup.
Services:api-gateway, account-service, transaction-service, notification-service
Required deps:spring-retry + spring-boot-starter-aop (both needed — AOP provides retry interception)
Config:fail-fast: true, max-attempts: 6, initial-interval: 1000ms, multiplier: 1.1, max-interval: 2000ms
7 API Gateway Default Port Changed 8080 → 8090 Phase 2 ✓ WORKING
⚠ Before — Problem
Standard gateway default port 8080 directly conflicts with banking-auth-api (Project 1) which also runs on 8080. Running both projects side-by-side — as expected in a portfolio demo — requires stopping one first. Defeats the demonstration purpose.
✓ After — Solution
GATEWAY_PORT env var defaults to 8090 in api-gateway.yml. banking-auth-api stays on 8080 unchanged. Both projects run simultaneously on the same machine with no port conflicts. Still overridable: set GATEWAY_PORT=8099
Services:api-gateway
Conflict avoided with:banking-auth-api (Project 1) on port 8080
New default port:8090 — still overridable via GATEWAY_PORT env var
10 Startup Order: Config → Registry → Services → Gateway Phase 6 ✓ WORKING
⚠ Before — Problem
Services started out of order crash immediately: account-service can't reach config-server (not up yet) → Connection refused. api-gateway routes to services not yet registered in Eureka → 503 No instances available. Docker Compose depends_on by default only waits for container start, not service readiness.
✓ After — Solution
Docker Compose uses depends_on: condition: service_healthy with HTTP health checks. Each service declares a healthcheck pointing to /actuator/health. Enforced order: config-server → service-registry → account/transaction/notification → api-gateway. Local start-services.bat follows the same order with port-readiness polling.
Services:All 6
Mechanism:Docker Compose healthcheck + depends_on: condition: service_healthy
Health endpoints:config-server :8888/actuator/health → service-registry :8761/actuator/health → business services → gateway :8090/actuator/health
Service Design Decisions
5 Notification-Service: No Web Server (Pure Kafka Consumer Process) Phase 5 ✓ WORKING
⚠ Before — Problem
Adding spring-boot-starter-web starts an embedded Tomcat server that serves zero REST endpoints. Wastes ~50MB heap, binds an unnecessary port, exposes an empty service on the network. Requires Eureka registration despite having nothing to register.
✓ After — Solution
spring-boot-starter-web excluded from POM entirely. Without web starter on classpath, Spring Boot auto-detects web-application-type=NONE. No Tomcat started. No port bound. Pure JVM background process consuming Kafka events. No Eureka registration needed — nothing routes to this service.
Services:notification-service only
Trade-off:No HTTP health endpoint — monitor via Kafka consumer group lag in Kafdrop UI (:9000)
Eureka:Not registered — consumers don't need discovery; no service calls notification-service
6 Explicit Kafka Topic Creation via @Bean NewTopic (KafkaTopicConfig) Phase 3 ✓ WORKING
⚠ Before — Problem
Relying on Kafka's auto.create.topics.enable=true creates topics with defaults: 1 partition, replication factor 1. No control over partitioning strategy. In production, this setting is typically disabled — topics must be pre-created. Missing topics cause silent message drops with no error until consumed.
✓ After — Solution
KafkaTopicConfig class with one @Bean NewTopic per topic. KafkaAdmin automatically creates or verifies topics on startup. Explicit partition count (3) and replication factor (1 dev / 3 prod) per topic. Clean, production-aligned approach — safe even when auto-creation is disabled.
Services:account-service (producer config), transaction-service (producer config)
Topics:account.created, account.updated, transaction.initiated, transaction.completed, notification.DLT
Class:KafkaTopicConfig.java in account-service/config/ and transaction-service/config/
Kafka & Testing Decisions
8 @EmbeddedKafka with Port 0 (Random) for Test Isolation Phase 7 ✓ WORKING
⚠ Before — Problem
Without explicit port configuration, @EmbeddedKafka may bind on a fixed port (e.g. 9092). If Docker Kafka is running on 9092 simultaneously, the test fails with Address already in use: 9092. Tests cannot run alongside a live Kafka environment.
✓ After — Solution
@EmbeddedKafka(partitions=1, brokerProperties={"listeners=PLAINTEXT://localhost:0"}) — port 0 instructs the OS to assign a random free port. Test config wires broker address via ${spring.embedded.kafka.brokers}. Tests are fully isolated from any running Kafka instance. CI always passes.
Services:account-service test, transaction-service test, notification-service test
Dependency:spring-kafka-test (test scope) — already in all 3 service POMs
Port strategy:0 = OS-assigned random — guaranteed no conflict with real Kafka on 9092
9 @RetryableTopic with Dead Letter Topic (DLT) — Non-Blocking Retry Phase 5 ✓ WORKING
⚠ Before — Problem
Without retry, a single transient failure (NPE, downstream timeout, deserialization error) causes the Kafka consumer to skip the message permanently — silent data loss. Manual DLQ requires writing a custom SeekToCurrentErrorHandler and complex DeadLetterPublishingRecoverer configuration.
✓ After — Solution
@RetryableTopic(attempts="3", backoff=@Backoff(delay=1000)) on consumer method. Spring Kafka retries 3× with 1s backoff automatically (non-blocking — other messages still processed). After 3 failures, message is routed to notification.DLT topic for inspection via Kafdrop. Zero manual DLQ plumbing — fully Spring-managed.
Services:notification-service
DLT topic:notification.DLT — inspectable in Kafdrop UI at http://localhost:${KAFDROP_PORT:9000}
Retry config:3 attempts, 1s backoff — override via @Backoff parameters
Non-blocking:Failed message retry goes to a retry topic, not blocking the main partition
