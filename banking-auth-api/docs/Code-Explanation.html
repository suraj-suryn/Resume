Banking Auth API — Complete Code Explanation

Every annotation · Every library · Every decision — explained from scratch with your own code

1. Project Overview

This project is a REST API for a banking system. Its main job is:

  • Let users register and login
  • Issue JWT tokens (like a digital pass) on login
  • Let authenticated users open accounts, transfer money, view history
  • Protect every endpoint so only the right person can access it

Request Flow — What Happens When You Call an API

Example: POST /api/accounts (create a bank account)

🌐 Browser / Postman  →  POST /api/accounts  [Authorization: Bearer eyJhbG...]
▼
LAYER 1AuditLoggingFilter  @Order(1) — runs FIRST
  • Logs: [AUDIT-IN] POST /api/accounts  IP=127.0.0.1
  • Stores start time for response duration calculation
▼
LAYER 2JwtAuthFilter  OncePerRequestFilter
  • Reads Authorization: Bearer eyJhbG... header
  • Calls JwtService.isTokenValid(token)
  • If valid → extracts username, sets identity in SecurityContext
  • If missing/invalid → continues as anonymous user (Security will block next)
▼
LAYER 3Spring Security  SecurityConfig rules
  • Is /api/accounts in permitAll()?  NO
  • Is user authenticated?  YES (JWT was valid)
  • Does user have required role (USER)?  YES
  • → ALLOW — request passes through
▼
LAYER 4AccountController  @RestController
  • @PostMapping("/api/accounts") method is invoked
  • Bean Validation runs on @RequestBody — rejects if fields missing
  • Calls AccountService.createAccount(...)
▼
LAYER 5AccountServiceImpl  @Service
  • Business logic: generate unique account number
  • Creates Account entity, sets owner, type, balance
  • Calls AccountRepository.save(account)
▼
LAYER 6AccountRepository  Spring Data JPA
  • Hibernate translates to SQL: INSERT INTO accounts (...)
  • Executes against H2 (dev) or PostgreSQL (prod)
  • Returns saved entity with generated id
▼
✅ Response: 201 Created  {"success":true,"data":{"id":1,"accountNumber":"ACC-...",...}}  |  AuditLoggingFilter logs: [AUDIT-OUT] 201 (45ms)

Swagger UI — What You Saw in the Browser

URL: localhost:8080/swagger-ui/index.html

Swagger is an interactive documentation tool. It reads your code's annotations and auto-generates a web page where you can test every API endpoint without Postman.

The "Available Authorizations" popup appeared because you clicked Authorize. You paste your JWT token there, and Swagger adds Authorization: Bearer <token> to every request automatically.

3. Core Spring Annotations

@SpringBootApplication

@SpringBootApplication
public class BankingAuthApiApplication {
    public static void main(String[] args) {
        SpringApplication.run(BankingAuthApiApplication.class, args);
    }
}

What it does: This is actually 3 annotations in 1:

  • @SpringBootConfiguration — marks this as a configuration class
  • @EnableAutoConfiguration — tells Spring Boot to auto-configure everything (sees H2 on classpath → auto-configures H2 datasource)
  • @ComponentScan — scans the current package and sub-packages for @Component, @Service, @Repository, @Controller classes and registers them as Spring beans

If you replaced it with just @Configuration: Spring would start but nothing would be auto-configured — no embedded server, no datasource, no security.

@Configuration and @Bean

@Configuration   // This class can define beans
@EnableWebSecurity
public class SecurityConfig {

    @Bean   // Spring will call this method and manage the returned object
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) { ... }
}

@Configuration — Tells Spring: "This class contains bean definitions." Spring reads all @Bean methods inside it at startup.

@Bean — Tells Spring: "Call this method once, store the result, and give it to anyone who asks for a PasswordEncoder." This is called Dependency Injection.

Real-world analogy: @Configuration is like a factory. @Bean is like a production line in that factory. When you need a PasswordEncoder, you don't say "new BCryptPasswordEncoder()" — you ask the factory and it gives you the one it already made.

Why not just use "new" everywhere? If you use "new", you create a new object every time. With @Bean, Spring creates one object and reuses it everywhere (Singleton by default). Also, Spring can inject other dependencies into the object automatically.

Stereotype Annotations — @Service, @Repository, @RestController, @Component

These are all variations of @Component. They all tell Spring "create an instance of this class and manage it." The difference is semantic:

AnnotationUsed onWhat it signalsExtra behaviour
@ComponentAny classGeneric Spring-managed beanNone
@ServiceService classesBusiness logic layerNone (just clarity)
@RepositoryDAO classesData access layerAuto-translates DB exceptions to Spring's DataAccessException
@RestControllerController classesHTTP request handler= @Controller + @ResponseBody (auto-converts return value to JSON)
@ConfigurationConfig classesBean factoryMethods annotated with @Bean are intercepted by Spring proxy
What is a "Bean"?
A bean is simply a Java object that Spring creates and manages. When Spring starts, it scans for all classes annotated with the above annotations, creates one instance of each, and puts them in a container called the Application Context. When any class needs another class, Spring injects it automatically.

@Value — Reading from application.properties

@Value("${app.jwt.secret}")
private String jwtSecret;

@Value("${app.jwt.expiration-ms}")
private long jwtExpirationMs;

In application.properties: app.jwt.secret=banking-auth-api-secret-key...

What @Value does: At startup, Spring reads the properties file and injects the value directly into the field. You never hardcode secrets in Java code.

If you hardcoded it: private String secret = "my-secret"; — the secret is in your source code and gets committed to git. Anyone who sees your repo sees your secret. Very bad for security.

In production: application-prod.properties uses ${JWT_SECRET} — this reads from environment variables, so the secret is never in any file.

4. JPA & Entities — Mapping Java to Database

JPA (Java Persistence API) lets you work with database records as Java objects. You never write SQL manually for basic operations.

@Entity and @Table

@Entity
@Table(name = "users")
public class User {
    ...
}

@Entity — Tells Hibernate (the JPA implementation): "This Java class corresponds to a database table." Hibernate will create a users table automatically (because spring.jpa.hibernate.ddl-auto=update).

@Table(name = "users") — The table name. If you omit this, Hibernate uses the class name ("user"). But "USER" is a reserved word in SQL, so we specify "users".

If you removed @Entity: Hibernate ignores this class entirely. No table, no queries — saving a User throws an exception.

@Id and @GeneratedValue

@Id
@GeneratedValue(strategy = GenerationType.IDENTITY)
private Long id;

@Id — This field is the primary key of the table.

@GeneratedValue(strategy = IDENTITY) — The database auto-generates the ID (like AUTO_INCREMENT in MySQL, SERIAL in PostgreSQL). You don't set it — the DB does.

Other strategies:

  • IDENTITY — DB generates it (most common, works with PostgreSQL/MySQL)
  • SEQUENCE — Uses a DB sequence object (best for PostgreSQL performance)
  • AUTO — Hibernate decides which strategy to use
  • TABLE — Uses a separate table to track IDs (slowest, rarely used)

@Column and @Enumerated

@Column(unique = true, nullable = false, length = 50)
private String username;

@Column(precision = 19, scale = 4)
private BigDecimal balance;

@Enumerated(EnumType.STRING)
private Role role;

@Column(unique=true) — Creates a UNIQUE constraint in the DB. Two users can't have the same username.

@Column(nullable=false) — Creates a NOT NULL constraint. Username can never be null.

BigDecimal(precision=19, scale=4) — For money: 19 total digits, 4 decimal places. Example: 123456789012345.6789. Never use float or double for money — they have rounding errors (0.1 + 0.2 = 0.30000000000000004).

@Enumerated(EnumType.STRING) — Stores the enum as a string ("USER", "ADMIN") in the DB, not as a number (0, 1). String is safer — if you reorder the enum, numbers would change meaning.

@OneToMany and @ManyToOne — Relationships

// In User.java — one user has many accounts
@OneToMany(mappedBy = "user", cascade = CascadeType.ALL, fetch = FetchType.LAZY)
private List<Account> accounts;

// In Account.java — many accounts belong to one user
@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "user_id")
private User user;

FetchType.LAZY — When you load a User from DB, Hibernate does NOT immediately load all their accounts. It only loads accounts when you actually access user.getAccounts(). This is performance-efficient — you don't load data you don't need.

FetchType.EAGER — Loads related data immediately, every time. Risky — if a user has 1000 transactions, you load all 1000 even when you just need the username.

CascadeType.ALL — If you delete a User, Hibernate automatically deletes all their Accounts too. If you save a User with new Accounts in the list, Hibernate saves the Accounts too.

@JoinColumn(name="user_id") — Creates a foreign key column named "user_id" in the accounts table.

@PrePersist — Auto-set values before saving

@PrePersist
protected void onCreate() {
    createdAt = LocalDateTime.now();
    active = true;
}

What it does: This method runs automatically, right before Hibernate saves a new record to the database. You never need to call it manually.

Why use it: If you set createdAt in the constructor or service, someone could accidentally pass a different date. With @PrePersist, the DB layer always sets the correct timestamp — impossible to override from outside.

Similar annotations: @PreUpdate (runs before UPDATE), @PostLoad (runs after SELECT), @PostPersist (runs after INSERT).

Repositories — Spring Data JPA Magic

public interface UserRepository extends JpaRepository<User, Long> {
    Optional<User> findByUsername(String username);
    boolean existsByEmail(String email);
}

JpaRepository<User, Long> — By extending this, your repository gets 18+ methods for free:

  • save(user) — INSERT or UPDATE
  • findById(1L) — SELECT WHERE id=1
  • findAll() — SELECT * FROM users
  • delete(user) — DELETE
  • count() — SELECT COUNT(*)
  • ...and more

findByUsername(String username) — You don't write any SQL! Spring Data reads the method name and generates: SELECT * FROM users WHERE username = ?

existsByEmail(String email) — Generates: SELECT COUNT(*) > 0 FROM users WHERE email = ?

Method naming rules: findBy / existsBy / countBy / deleteBy + field name. You can chain: findByUsernameAndActive(String, boolean)

You never create an implementation class. Spring creates a proxy at runtime that implements all these methods.

@Query and Pageable — Custom Queries

// In TransactionRepository.java
@Query("SELECT t FROM Transaction t WHERE t.account.id = :accountId ORDER BY t.createdAt DESC")
Page<Transaction> findByAccountId(@Param("accountId") Long accountId, Pageable pageable);

@Query — For when the method name convention isn't enough. You write JPQL (Java Persistence Query Language) — like SQL but uses Java class/field names, not table/column names.

JPQL vs SQL:

  • SQL: SELECT * FROM transactions WHERE account_id = 1
  • JPQL: SELECT t FROM Transaction t WHERE t.account.id = 1

Pageable — Enables pagination. Caller passes PageRequest.of(0, 10) — page 0, 10 records. The DB query becomes: SELECT ... LIMIT 10 OFFSET 0. Returns a Page<T> object containing the records AND total count.

Why pagination: If a user has 10,000 transactions, you don't want to load all 10,000 at once. Pagination fetches 10 at a time.

5. Lombok — Eliminating Boilerplate Code

Lombok is an annotation processor. It runs at compile time and generates Java code for you. Your source file is short; the compiled .class file has all the methods.

@Builder and @Data

@Data           // generates: getters + setters + toString + equals + hashCode
@Builder        // generates: builder pattern
@Entity
public class Account {
    private Long id;
    private String accountNumber;
    private BigDecimal balance;
}

Without Lombok, you'd write:

// 50+ lines of boilerplate that Lombok generates for you:
public Long getId() { return id; }
public void setId(Long id) { this.id = id; }
public String getAccountNumber() { return accountNumber; }
public void setAccountNumber(String n) { this.accountNumber = n; }
// ... and equals, hashCode, toString, builder class...

With @Builder, you can construct objects like this:

Account account = Account.builder()
    .accountNumber("ACC123")
    .balance(BigDecimal.ZERO)
    .accountType(AccountType.SAVINGS)
    .build();
Why use builder pattern instead of constructor? If a class has 8 fields, a constructor call is: new Account(null, "ACC", null, ZERO, null, null, true, null) — impossible to tell which null is which. Builder is readable and you only set what you need.

Constructor Annotations

AnnotationGeneratesUsed when
@NoArgsConstructorpublic Account() {}JPA requires a no-arg constructor to instantiate entities from DB rows
@AllArgsConstructorConstructor with all fieldsUsed internally by @Builder
@RequiredArgsConstructorConstructor with final fields onlyUsed on @Service/@Configuration — Spring injects dependencies via constructor (best practice)
@RequiredArgsConstructor + final fields = Constructor Injection
In SecurityConfig: private final JwtAuthFilter jwtAuthFilter;
Lombok generates: public SecurityConfig(JwtAuthFilter jwtAuthFilter) { this.jwtAuthFilter = jwtAuthFilter; }
Spring sees this constructor and automatically provides the JwtAuthFilter bean.
This is preferred over @Autowired field injection because: it makes dependencies explicit, enables immutability (final), and makes testing easier.

6. DTOs — Data Transfer Objects

Why DTOs? Why not use Entity classes directly?

❌ Without DTO (expose Entity directly)

// If you return User entity:
{
  "id": 1,
  "username": "suraj",
  "passwordHash": "$2a$10$...", // EXPOSED!
  "accounts": [...], // too much data
  "role": "USER"
}

Security risk: password hash exposed. Performance issue: loads all related data.

✅ With DTO (return only what's needed)

// TokenResponse DTO returns:
{
  "accessToken": "eyJ...",
  "tokenType": "Bearer",
  "username": "suraj",
  "role": "USER"
}

Only safe, relevant fields. No internal data exposed.

Bean Validation on DTOs

public class RegisterRequest {

    @NotBlank(message = "Username is required")
    @Size(min = 3, max = 50)
    private String username;

    @Email(message = "Must be a valid email")
    private String email;

    @Size(min = 6, message = "Password must be at least 6 characters")
    private String password;
}

In the controller, @Valid triggers validation:

@PostMapping("/register")
public ResponseEntity register(@Valid @RequestBody RegisterRequest request) {

If validation fails, Spring automatically throws MethodArgumentNotValidException before the method even runs. Your GlobalExceptionHandler catches it and returns a 400 with field-level errors:

{ "success": false, "data": { "username": "must not be blank", "password": "size must be between 6 and ..." }}

If you removed @Valid: Invalid data would reach your service layer. A user could register with empty username, 1-character password, or invalid email.

@JsonInclude(NON_NULL)

@JsonInclude(JsonInclude.Include.NON_NULL)
public class ApiResponse<T> {
    private boolean success;
    private String message;
    private T data;    // can be null for error responses
}

Without this: {"success":false,"message":"Not found","data":null} — data:null is included

With @JsonInclude(NON_NULL): {"success":false,"message":"Not found"} — null fields are omitted, cleaner response

7. Spring Security

What Spring Security Does

Spring Security sits between the HTTP request and your controllers like a bouncer at a club. It runs its logic before your code runs. You configure rules like: "Anyone can enter /api/auth/**. Only people with a valid JWT can enter /api/accounts/**."

SecurityFilterChain — The Rulebook

@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http
        .csrf().disable()                    // ← explained below
        .headers().frameOptions().disable()  // ← for H2 console iframes
        .and()
        .exceptionHandling()
            .authenticationEntryPoint(
                new HttpStatusEntryPoint(HttpStatus.UNAUTHORIZED))  // ← returns 401 JSON
        .and()
        .authorizeHttpRequests()
            .antMatchers("/api/auth/**").permitAll()        // ← no JWT needed
            .antMatchers("/h2-console/**").permitAll()
            .antMatchers("/actuator/health").permitAll()
            .antMatchers("/swagger-ui/**").permitAll()
            .anyRequest().authenticated()                    // ← everything else needs JWT
        .and()
        .sessionManagement()
            .sessionCreationPolicy(SessionCreationPolicy.STATELESS)  // ← no sessions
        .and()
        .addFilterBefore(jwtAuthFilter, UsernamePasswordAuthenticationFilter.class);

    return http.build();
}

CSRF Disabled — Why?

CSRF (Cross-Site Request Forgery): An attack where a malicious website tricks your browser into making requests to your bank while you're logged in. Protection works by requiring a secret token in every form submission — only your real site knows it.

Why we disable it: CSRF only matters for cookie-based sessions. Our API is stateless JWT-based — the client (mobile app, React app) explicitly adds the Authorization header with the JWT. A malicious website can't add that header. So CSRF protection is not needed and would just add complexity.

If you enabled CSRF: Every POST/PUT/DELETE would need an extra CSRF token parameter. Postman and Swagger would stop working without extra configuration.

STATELESS Sessions — Why?

Session-based (traditional)

Server creates a session, stores user data in memory or Redis. Client gets a session cookie. Every request: server looks up session in storage.

Problem: Can't scale horizontally easily. If you have 3 servers, request might go to a server that doesn't have the session.

JWT / STATELESS (our approach)

Server issues a JWT containing all user info. Client sends JWT with every request. Server verifies the signature — no storage lookup needed.

Advantage: Any server can verify any token. Scales perfectly. No shared session storage needed.

@PreAuthorize — Method-Level Security

@PreAuthorize("isAuthenticated()")
@PostMapping
public ResponseEntity createAccount(@AuthenticationPrincipal UserDetails userDetails) {...}

// Or check role:
@PreAuthorize("hasRole('ADMIN')")
public ResponseEntity adminEndpoint() {...}

@EnableMethodSecurity in SecurityConfig enables this. @PreAuthorize runs before the method body. If the condition is false, Spring throws AccessDeniedException (403 Forbidden) before your code executes.

Spring Expression Language (SpEL): isAuthenticated(), hasRole('ADMIN'), hasAnyRole('USER','MANAGER'), principal.username == #username

Why ROLE_ prefix: Spring Security internally stores roles as "ROLE_USER", "ROLE_ADMIN". When you write hasRole('USER'), Spring checks for "ROLE_USER". Your CustomUserDetailsService adds this prefix: "ROLE_" + user.getRole().

BCrypt — Password Hashing

// When registering:
user.setPasswordHash(passwordEncoder.encode(request.getPassword()));

// When logging in (Spring Security does this internally):
passwordEncoder.matches(rawPassword, storedHash)  // returns true/false

NEVER store plain text passwords. BCrypt produces something like: $2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy

Key properties of BCrypt:

  • Same password always produces a different hash (due to random salt)
  • One-way — you can't reverse it back to the password
  • Slow by design — makes brute-force attacks expensive (cost factor 10 = 2^10 = 1024 rounds)

If you used MD5 or SHA256: These are fast hashing algorithms — attackers can test billions of passwords per second. BCrypt is intentionally slow.

UserDetails and UserDetailsService

@Service
public class CustomUserDetailsService implements UserDetailsService {

    @Override
    public UserDetails loadUserByUsername(String username) {
        User user = userRepository.findByUsername(username)
            .orElseThrow(() -> new UsernameNotFoundException("User not found"));

        return org.springframework.security.core.userdetails.User
            .withUsername(user.getUsername())
            .password(user.getPasswordHash())
            .authorities("ROLE_" + user.getRole())
            .disabled(!user.isActive())
            .build();
    }
}

UserDetailsService is a Spring Security interface with one method. Spring Security calls loadUserByUsername() during login to get the user's details and verify the password.

UserDetails is another Spring Security interface representing the logged-in user — contains username, hashed password, roles, and account status flags.

The two User classes: Note we use com.suraj.banking.auth.entity.User (your JPA entity) to load from DB, then convert to org.springframework.security.core.userdetails.User (Spring's interface). These are two different classes with the same name.

8. JWT — JSON Web Token

What is JWT?

A JWT is a digitally signed string that carries information. When you log in, the server creates a JWT, signs it with a secret key, and sends it back. On every future request, you send the JWT. The server verifies the signature to confirm it hasn't been tampered with — no DB lookup needed.

Real-world analogy: A JWT is like a signed cheque. Anyone can read the amount (it's not encrypted by default), but only the bank can create one with the valid signature. You can't forge it without the bank's secret stamp.

Token Structure — Header.Payload.Signature

eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJzdXJhajEyMyIsInJvbGUiOiJVU0VSIiwiaWF0IjoxNzE4ODc1MjAwLCJleHAiOjE3MTg4Nzg4MDB9.abc123signature

Part 1 (Header) — decoded:
{ "alg": "HS256" }

Part 2 (Payload/Claims) — decoded:
{
  "sub": "suraj123",      // subject = username
  "role": "USER",         // custom claim we added
  "iat": 1718875200,      // issued at (Unix timestamp)
  "exp": 1718878800       // expires at (iat + 1 hour)
}

Part 3 (Signature):
HMACSHA256(base64(header) + "." + base64(payload), secret)
Important: JWT payload is base64-encoded, NOT encrypted. Anyone can decode and read it. Never put sensitive data (passwords, credit card numbers) in a JWT payload. We only put username and role — both are non-sensitive.

JwtService — Token Creation & Validation

public String generateAccessToken(UserDetails userDetails, String role) {
    return Jwts.builder()
        .setSubject(userDetails.getUsername())     // "sub" claim
        .claim("role", role)                      // custom claim
        .setIssuedAt(new Date())                   // "iat" claim
        .setExpiration(new Date(System.currentTimeMillis() + jwtExpirationMs))
        .signWith(getSigningKey(), SignatureAlgorithm.HS256)
        .compact();                                // builds the string
}

private Key getSigningKey() {
    // Converts the secret string to a cryptographic key
    return Keys.hmacShaKeyFor(Decoders.BASE64.decode(jwtSecret));
}

HS256 (HMAC-SHA256): A symmetric signing algorithm — same secret key is used to sign and to verify. Simple and fast.

Alternative — RS256 (RSA): Asymmetric — private key signs, public key verifies. Used when multiple services need to verify tokens without sharing the secret (e.g., microservices). More complex. HS256 is fine for a single-service API.

Keys.hmacShaKeyFor(): HS256 requires a key of at least 256 bits (32 bytes). This method enforces that requirement — throws exception if key is too short.

9. Filters — Intercepting Every HTTP Request

What is a Filter?

A Servlet Filter sits in front of your entire application. Every HTTP request passes through all filters before reaching your controller. Filters can read/modify the request, read/modify the response, or block the request entirely.

Think of it as: Security checkpoints at an airport. Every passenger (request) goes through customs (filter chain) before boarding the plane (controller).

OncePerRequestFilter

public class JwtAuthFilter extends OncePerRequestFilter {
    @Override
    protected void doFilterInternal(...) { ... }
}

Why OncePerRequestFilter, not plain Filter? In some scenarios (like error forwarding), the Servlet filter chain can be called multiple times for the same original request. OncePerRequestFilter guarantees your logic runs exactly once per request, no matter what.

JwtAuthFilter — Step by Step

@Component
public class JwtAuthFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(
            HttpServletRequest request,
            HttpServletResponse response,
            FilterChain filterChain) throws Exception {

        // Step 1: Read the Authorization header
        String authHeader = request.getHeader("Authorization");

        // Step 2: Check it starts with "Bearer "
        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            filterChain.doFilter(request, response); // no token, pass through
            return;
        }

        // Step 3: Extract token (everything after "Bearer ")
        String token = authHeader.substring(7);

        // Step 4: Extract username from token
        String username = jwtService.extractUsername(token);

        // Step 5: If valid and not already authenticated
        if (username != null &&
            SecurityContextHolder.getContext().getAuthentication() == null) {

            UserDetails userDetails = userDetailsService.loadUserByUsername(username);

            if (jwtService.isTokenValid(token, userDetails)) {
                // Step 6: Create authentication object
                UsernamePasswordAuthenticationToken authToken =
                    new UsernamePasswordAuthenticationToken(
                        userDetails, null, userDetails.getAuthorities());

                // Step 7: Store in SecurityContext — "this user is authenticated"
                SecurityContextHolder.getContext().setAuthentication(authToken);
            }
        }

        // Step 8: Continue to next filter / controller
        filterChain.doFilter(request, response);
    }
}

SecurityContextHolder is like a thread-local storage. It holds who is currently logged in for this specific request's thread. When your controller calls @AuthenticationPrincipal UserDetails u, Spring reads from SecurityContextHolder.

Why not throw exception if token is missing? If there's no token, the filter just passes the request through as "anonymous". The SecurityConfig's .anyRequest().authenticated() rule will then reject it with 401. Separation of concerns — filter authenticates, SecurityConfig authorizes.

AuditLoggingFilter

@Component
@Order(1)  // runs FIRST in the filter chain
public class AuditLoggingFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(...) {
        long start = System.currentTimeMillis();
        String ip = getClientIp(request);

        log.info("[AUDIT-IN] {} {} IP={}", request.getMethod(), request.getRequestURI(), ip);

        filterChain.doFilter(request, response);  // run the rest of the chain

        // This line runs AFTER the controller responds:
        log.info("[AUDIT-OUT] status={} duration={}ms",
            response.getStatus(), System.currentTimeMillis() - start);
    }
}

@Order(1) — Lowest number = highest priority. This filter runs before all others, so it can measure the total time including all other filters and the controller.

X-Forwarded-For: When your app sits behind a proxy/load balancer, the real client IP is in this header. If it's missing, fall back to request.getRemoteAddr().

10. Exception Handling

Exception Hierarchy in This Project

RuntimeException (Java)
    └── AccountNotFoundException     → 404 Not Found
    └── InsufficientFundsException   → 422 Unprocessable Entity
    └── DuplicateUserException       → 409 Conflict
    └── InvalidTokenException        → 401 Unauthorized
    └── UnauthorizedAccessException  → 403 Forbidden

Spring's exceptions (handled too):
    ├── BadCredentialsException      → 401 Unauthorized (wrong password)
    ├── AccessDeniedException        → 403 Forbidden
    └── MethodArgumentNotValidException → 400 Bad Request (validation)

Why extend RuntimeException, not Exception? Checked exceptions (extends Exception) must be declared in method signatures with throws and must be caught. Unchecked/runtime exceptions don't need to be declared. For business exceptions, unchecked is cleaner — you don't want to pollute every method signature.

@RestControllerAdvice — Centralized Error Handling

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(AccountNotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)
    public ApiResponse<Void> handleNotFound(AccountNotFoundException ex) {
        return ApiResponse.error(ex.getMessage());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public ApiResponse<Map<String, String>> handleValidation(
            MethodArgumentNotValidException ex) {

        Map<String, String> errors = ex.getBindingResult()
            .getFieldErrors().stream()
            .collect(Collectors.toMap(
                FieldError::getField,
                FieldError::getDefaultMessage));

        return ApiResponse.error("Validation failed", errors);
    }
}

@RestControllerAdvice = @ControllerAdvice + @ResponseBody. It's a global exception handler for all controllers. Without it, you'd have to write try-catch in every single controller method.

Before:

// Every controller method would need this:
try {
    return accountService.getAccount(id);
} catch (AccountNotFoundException e) {
    return ResponseEntity.notFound().build();
}

After: Controller is clean. Exception is thrown anywhere, GlobalExceptionHandler catches it, returns consistent JSON.

    }
}

ResponseEntity<ApiResponse<TokenResponse>> — Nested generics. ResponseEntity lets you control the HTTP status code. ApiResponse is your wrapper. TokenResponse is the actual data. This gives full control: status + headers + body.

Why not just return TokenResponse directly? You couldn't set 201 vs 200, or add custom headers. ResponseEntity is more explicit.

@RestController

}

12. Service Layer

Interface + Implementation Pattern

// Interface:
public interface AccountService {
    AccountResponse createAccount(String username, AccountType type);
    List<AccountResponse> getMyAccounts(String username);
}

// Implementation:
@Service
public class AccountServiceImpl implements AccountService { ... }

Why interface + impl?

  • Testing: In tests, you can create a mock that implements the interface without touching the real implementation
  • Flexibility: You could swap AccountServiceImpl with AccountServiceV2Impl without changing the controller
  • Contracts: The interface defines what it does; the impl defines how. Controllers only depend on the interface
  • Spring proxy: Spring wraps your @Service in a proxy to add @Transactional behaviour — this requires an interface

@Transactional — Atomic Operations

@Transactional
public TransactionResponse transfer(
        Long fromAccountId, String username, TransferRequest request) {

    Account fromAccount = accountRepository.findById(fromAccountId)...;
    Account toAccount = accountRepository.findByAccountNumber(...)...;

    // Check balance
    if (fromAccount.getBalance().compareTo(request.getAmount()) < 0) {
        throw new InsufficientFundsException("Insufficient funds");
    }

    // Debit source account
    fromAccount.setBalance(fromAccount.getBalance().subtract(request.getAmount()));

    // Credit target account
    toAccount.setBalance(toAccount.getBalance().add(request.getAmount()));

    // Create transaction record
    Transaction txn = Transaction.builder()...build();

    accountRepository.save(fromAccount);
    accountRepository.save(toAccount);
    transactionRepository.save(txn);
    return mapToResponse(txn);
}

@Transactional = all or nothing.

Scenario: Balance is debited from source, then server crashes before crediting target. Without @Transactional, money disappears.

With @Transactional: If anything throws an exception, all changes are rolled back — both accounts return to original balances. Either the full transfer succeeds, or nothing changes.

How it works: Spring wraps the method in a database transaction. On success → COMMIT. On exception → ROLLBACK.

Default behaviour: Rolls back on RuntimeException. Does NOT roll back on checked exceptions. For checked exceptions, use: @Transactional(rollbackFor = Exception.class)

13. Testing — JUnit 5 + Mockito

Mockito Annotations

@ExtendWith(MockitoExtension.class)
class AuthServiceTest {

    @Mock
    private UserRepository userRepository;   // fake — does not hit DB

    @Mock
    private PasswordEncoder passwordEncoder;  // fake

    @InjectMocks
    private AuthServiceImpl authService;     // real class, fake dependencies injected
}

@Mock — Creates a fake version of this class. By default, all methods return null/0/false. You program specific behaviour with when(...).thenReturn(...).

@InjectMocks — Creates the real AuthServiceImpl and injects all @Mock objects into it. Your real code runs, but with fake dependencies — so tests are fast and don't need a database.

Why mock the repository? You don't want unit tests to depend on a real database. Tests should be fast (milliseconds), isolated, and reproducible. A real DB connection is slow and flaky.

Test Examples

@Test
void shouldRegisterNewUser() {
    // ARRANGE — set up mocks
    when(userRepository.existsByUsername("testuser")).thenReturn(false);
    when(userRepository.existsByEmail("test@test.com")).thenReturn(false);
    when(passwordEncoder.encode("password123")).thenReturn("$2a$hashed");
    when(userRepository.save(any())).thenReturn(testUser);

    // ACT — call the real method
    TokenResponse result = authService.register(registerRequest);

    // ASSERT — verify the result
    assertThat(result).isNotNull();
    assertThat(result.getUsername()).isEqualTo("testuser");
    verify(userRepository, times(1)).save(any(User.class));
}

@Test
void shouldThrowWhenUsernameAlreadyExists() {
    when(userRepository.existsByUsername(anyString())).thenReturn(true);

    // This lambda should throw DuplicateUserException:
    assertThrows(DuplicateUserException.class,
        () -> authService.register(registerRequest));
}

AAA Pattern (Arrange-Act-Assert): Every test should follow this structure.

verify(userRepository, times(1)).save(...) — Verifies that the mock's save() method was called exactly once. Ensures your code actually persisted the user.

assertThrows() — Verifies that the code throws the expected exception. If no exception is thrown, the test fails.

14. Interview Q&A

Q: What is Spring Boot and how is it different from Spring Framework?
Spring Framework is a large collection of modules for building Java applications. Spring Boot is built on top of it and adds: auto-configuration (no XML config needed), embedded server (Tomcat built-in), opinionated defaults (sensible configs out of the box), and starter dependencies that bundle commonly used libraries together.
Q: What is Dependency Injection and why use it?
Dependency Injection means a class doesn't create its own dependencies — it receives them from outside (from Spring). Example: AccountServiceImpl needs UserRepository. Instead of doing new UserRepository(), Spring creates the UserRepository and "injects" it via the constructor. Benefits: testable (you can inject mocks), loosely coupled (swap implementations easily), managed lifecycle (Spring handles creation/destruction).
Q: Why use JWT instead of session-based authentication?
JWT is stateless — the server doesn't store anything. Each token carries all needed information (username, role, expiry) and is self-verifiable via the signature. Sessions require server-side storage; if you have multiple servers, you need a shared session store (Redis). JWT scales horizontally without any shared state.
Q: What is the difference between @Controller and @RestController?
@RestController = @Controller + @ResponseBody. @ResponseBody means the return value is written directly to the HTTP response body (serialized to JSON by Jackson). Without @ResponseBody (plain @Controller), Spring would try to find a view template (like a JSP/Thymeleaf file) matching the returned string.
Q: Why is @Transactional important for fund transfers?
A transfer involves multiple DB operations: debit one account and credit another. If the second operation fails, the first must be undone. @Transactional wraps both in a single database transaction. If anything fails, the entire transaction rolls back — money is never lost in transit.
Q: What is CSRF and why did we disable it?
CSRF is an attack where a malicious site tricks your browser into making requests to your API using your cookies. Protection adds a token to forms that only your real site knows. We disabled it because our API uses JWT Bearer tokens in headers — browsers can't be tricked into adding custom headers cross-origin, so CSRF is not applicable to our stateless API.
Q: What is the difference between @Mock and @Spy in Mockito?
@Mock creates a completely fake object — all methods do nothing by default. @Spy wraps a real object — real methods run unless you override specific ones with when(). For unit tests where we want to test our service logic in isolation, @Mock is appropriate — we don't want real DB calls.
Q: Why use BigDecimal for money instead of double?
floating-point types (float, double) can't represent decimal numbers exactly. 0.1 + 0.2 = 0.30000000000000004. For money, rounding errors are unacceptable. BigDecimal stores numbers exactly with configurable precision and scale. Java's BigDecimal.add(), subtract(), multiply() guarantee exact results.
Q: What is the role of OncePerRequestFilter?
In a Servlet container, the filter chain can potentially be invoked multiple times for the same request (e.g., when an error causes a forward to an error page). OncePerRequestFilter guarantees the filter logic executes exactly once per HTTP request by tracking execution in a request attribute.
Q: What is the purpose of LAZY loading in JPA?
LAZY loading defers the database query for related entities until you actually access them. If you load 100 users with EAGER loading on their transactions, you'd execute 101 queries (N+1 problem). With LAZY, you only load transactions when you explicitly access user.getTransactions(), reducing unnecessary database work.
